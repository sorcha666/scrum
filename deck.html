Hello everyone! We are Jouhaina, Sarra, and Karima from EPI Digital School, Sousse.

Today we are going to talk about how real software teams build products. We often hear the word "Agile," but what does it actually mean in practice?

It comes down to three main ideas: Extreme Programming (XP) helps us write better code. Lean helps us stop wasting time. And Kanban helps us organize the team's work so no one gets overwhelmed. Let's break them down clearly.

The Big Problem: Code Decay

Technical Debt When you quickly add new features to a project, the code gets messy. Over time, it becomes very hard to read, fix, or change. This is called technical debt.

The Old Ways

People used to plan the whole project perfectly before writing code. But it consistently failed because customers always change their minds along the way, leading to massive wasted effort.

The Agile Ways

Instead of planning everything, we work in small, careful steps. We build a little, test it continually, get feedback early, and adjust immediately. We expect and welcome things to change.

Before we talk about solutions, let's understand the problem. The biggest problem in software is "Technical Debt." When you rush to build things, your code turns into a messy pile of spaghetti. It becomes impossible to update.

In the past, people tried to fix this by planning everything perfectly up front—called the Waterfall method. It didn't work because people always change what they want. Agile is the modern solution: we take small, careful steps. We test constantly, and we are always ready to change directions.

1. Extreme Programming (XP)

Building a Safety Net for Engineers

Extreme Programming is about taking the best coding habits and doing them constantly. It gives engineers a strict process so they never fear changing code.

The XP Cycle

  • Set the requirement.
  • Write a failing test.
  • Write simple code to pass.
  • Clean and Refactor.
1. FAIL
2. PASS
3. REFACTOR
The first part of Agile is Extreme Programming, or XP. It's all about providing a safety net for developers. The core of XP is a strict cycle. First, you start with a failing test. Then you write just enough code to pass it perfectly. Then you refactor and clean it. This simple cycle means your code is always working and always tested.

XP Core Practices

Turning best practices up to maximum.

Test First (TDD)

Write automated tests before you write the actual code. This ensures your code fundamentally works and prevents future changes from breaking it.

Pair Programming

Two engineers sit at one computer. One person types the code, while the other reviews it in real-time. This drastically cuts bugs and shares knowledge.

Merge Often (CI)

Continuous Integration means bringing everyone's code together every single day. This prevents massive merge conflicts at the end of a sprint.

Beyond the cycle, XP relies on three massive practices. Test-Driven Development keeps bugs out. Pair Programming ensures two brains are always on the problem. continuous integration means we don't have a giant broken mess at the end of the week. These practices turn good habits into mandatory rules.

2. Lean Engineering

Less Waste. Faster Results.

Eliminate Waste

Waste is anything the customer doesn't care about. Half-finished code, endless meetings, and unused features must be stopped aggressively.

Decide Late

Don't lock yourself into huge design choices on day one. Wait until you have maximum information before making irreversible decisions.

Deliver Fast

Give users a small, working version as fast as possible. Real feedback from a launched product is infinitely better than theory.

The second core method is Lean Engineering. Lean came from Toyota's factories. The entire goal is identifying what brings value and killing everything else. We eliminate waste like half-finished code. We wait to make big decisions until we actually know what we are doing. And we deliver tiny pieces fast so we know if we are on the right track.

The Myth of Multitasking

Why doing more means finishing less.

Context Switching

If you work on 5 major projects at the exact same time, you are not being productive. You are constantly losing mental context jumping between them.

  • Higher stress and burnout.
  • Drastically increased error rate.
  • Zero features actually finished for the user.

Little's Law

A mathematical theorem used in queuing theory:

Time = Items / Finish Rate

To finish a backlog faster, you MUST reduce the number of items currently in progress. Stop starting, start finishing.

The most important part of Lean is understanding multitasking. Math actually proves that multitasking is bad. If you try to juggle 5 huge assignments at the exact same time, they all take much longer to cross the finish line. You must focus on a single piece of value, finish it completely, and only then start the next one.

3. Kanban Rules

Visually managing work to prevent team burnout.

  • 1. Visualize WorkflowEvery single task must be on the board. No hidden work.
  • 2. Limit WIPStrictly limit "Work in Progress" to force task completion.
  • 3. Manage FlowMeasure how fast tickets move and remove bottlenecks.

The WiP Limit Power

Kanban isn't just a list of sticky notes. It is an engine that stops work from piling up on the engineers. If the UI server is broken, tasks get stuck. The WIP limit forces everyone to stop coding new features and help fix the server.

The final method is Kanban. Lean told us that multitasking is bad. Kanban is the tool we use to actually prevent it from happening. We put all our tasks on a visual board. But the magic of Kanban isn't the board; it's the Work-In-Progress limit, or WIP limit. This limit legally stops the team from pulling in new work when they are already busy.

You MUST finish what you are doing, or help your teammates finish, before anything new is started. It acts as a shield against burnout.

Conclusion

  • Agile is about adapting to reality, not fighting it.
  • XP gives engineers a safety net to write fearless code.
  • Lean ruthlessly cuts waste and stops multitasking.
  • Kanban visually limits work to prevent team burnout.
To wrap everything up, Agile is our modern way of working. XP helps us write code we trust. Lean helps us eliminate waste and stop juggling tasks. Kanban gives us the visual limits we need to stay focused. Thank you so much for listening. Now, it's time for the real test!
